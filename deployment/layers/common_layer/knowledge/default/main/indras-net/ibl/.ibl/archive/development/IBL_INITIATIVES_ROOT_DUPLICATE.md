## RIPPLE FRAMEWORK

**What is a Ripple?**
A Ripple is a gentle technical suggestion—a perturbation in the Field that the project may choose to integrate or ignore. Ripples are never mandates.

**Ripple Structure**:
1. **Observation**: What Aletheia signal was detected?
2. **Analysis**: What is the root cause of the friction?
3. **Proposal**: What is the suggested improvement?
4. **Options**: What are alternative approaches?
5. **Impact**: What are the consequences of each option?
6. **Authorization**: Awaiting explicit approval from Ground Zero

**Ripple Example**:
```
Ripple #1: Missing Error Handling in API Endpoint

Observation: Detected unhandled exceptions in /api/users endpoint
Analysis: Endpoint lacks try-catch blocks, causing 500 errors on invalid input
Proposal: Add comprehensive error handling with appropriate HTTP status codes
Options:
  A. Add try-catch with specific exception types (recommended)
  B. Add global error handler middleware

Impact:
  A. Improved reliability, better error messages, graceful degradation
  B. Centralized error handling, consistent responses across endpoints
  C. No change, continues to generate 500 errors
Authorization: Awaiting Ground Zero approval
```

## INITIATIVE DEPENDENCY GRAPH VISUALIZATION

**Principle**: Visualize initiative dependencies as a directed graph to detect cycles, identify critical paths, and recommend parallel work opportunities.

**Dependency Graph Structure**:
```
DEPENDENCY_GRAPH = {
  nodes: [initiative_id],
  edges: [
    {from: initiative_A, to: initiative_B, type: "blocks|enables|shares_resource"}
  ],
  metadata: {
    critical_path: [initiative_sequence],
    parallel_clusters: [[initiative_set]],
    cycle_warnings: [cycle_description]
  }
}
```

**Cycle Detection** (Circular Dependencies):
- **Detection Algorithm**: Depth-first search for back edges in dependency graph
- **Cycle Types**:
  1. **Direct Cycle**: Initiative A → Initiative B → Initiative A
  2. **Indirect Cycle**: Initiative A → B → C → A
  3. **Resource Cycle**: A and B both waiting for same shared resource
- **Cycle Warning Format**:
  ```
  ⚠️ CIRCULAR DEPENDENCY DETECTED
  
  Cycle: Initiative {A} → Initiative {B} → Initiative {C} → Initiative {A}
  
  Problem: Each initiative is blocked waiting for another in the cycle.
  No progress possible without breaking the cycle.
  
  Recommended Actions:
  1. Reorder initiative dependencies (break weakest link)
  2. Split one initiative to remove dependency
  3. Create temporary workaround to unblock cycle
  
  Suggested Break Point: {initiative_name} (weakest dependency)
  ```
- **Auto-Detection**: Run cycle detection whenever new dependency added
- **Cycle Prevention**: Warn before adding edge that would create cycle

**Critical Path Analysis**:
- **Critical Path Definition**: Longest dependency chain from start to completion
- **Path Calculation**: 
  ```
  FOR each initiative without dependencies (root nodes):
    Calculate longest path to leaf nodes
  
  critical_path = max(all_paths)
  critical_path_length = sum(initiative_durations on critical_path)
  ```
- **Critical Path Visualization**:
  ```
  CRITICAL PATH (Estimated Duration: {total_time})
  
  Initiative 0: Ingress (2 hours) →
  Initiative 3: Database Setup (8 hours) →
  Initiative 1: Auth Implementation (12 hours) →
  Initiative 2: API Development (16 hours) →
  Initiative 5: Deployment (4 hours)
  
  Total: 42 hours (critical path)
  
  ⚠️ Delays on critical path directly impact project completion time.
  ```
- **Bottleneck Identification**: Initiatives on critical path with longest duration
- **Optimization Suggestions**:
  - Break large critical path initiatives into smaller parallel tasks
  - Allocate more resources to critical path initiatives
  - Start critical path work early

**Parallel Work Recommendations**:
- **Parallelization Detection**: Find initiatives with no mutual dependencies
- **Parallel Cluster Identification**:
  ```
  PARALLEL WORK OPPORTUNITIES
  
  Cluster 1 (Can work simultaneously):
  - Initiative 4: UI Development
  - Initiative 6: Documentation
  - Initiative 7: Testing Framework
  
  Cluster 2 (After Initiative 3 completes):
  - Initiative 1: Auth
  - Initiative 2: API
  
  Recommendation: Start Cluster 1 immediately to maximize throughput.
  Estimated time savings: 12 hours (vs sequential execution)
  ```
- **Resource Conflict Detection**: 
  - Check if parallel initiatives modify same files
  - Warn about potential merge conflicts
  - Suggest coordination protocol
- **Parallelization Benefits**:
  ```
  Sequential Execution: {total_sequential_time}
  Parallel Execution: {total_parallel_time}
  Time Saved: {savings} ({percentage}%)
  ```

**Dependency Graph Visualization** (ASCII Art):
```
Example Multi-Initiative Graph:

    [0: Ingress]
         |
         ├─────────┬─────────┐
         ↓         ↓         ↓
    [3: DB]   [4: UI]   [6: Docs]  ← Parallel Cluster 1
         |         |
         ↓         ↓
    [1: Auth] [2: API]              ← Parallel Cluster 2
         |         |
         └────┬────┘
              ↓
         [5: Deploy]

Critical Path: 0 → 3 → 1 → 5 (marked with heavy arrows)
Parallel Opportunities: {4, 6}, {1, 2}
```

**Dependency Tracking**:
- **Auto-Update**: Recalculate graph when:
  - New initiative added
  - Dependency relationship changes
  - Initiative completes (remove from graph)
- **Dependency Types**:
  - **Hard Dependency** (blocks): Initiative B cannot start until A completes
  - **Soft Dependency** (enables): B can start but will be easier after A
  - **Resource Dependency** (shares): A and B need same file/component
- **Dependency Strength**: Rate 0.0 (weak) to 1.0 (strong)
  - Strong dependencies (≥0.8): Must respect, cannot parallelize
  - Weak dependencies (<0.5): Can potentially work around

**Graph Metrics**:
- **Total Initiatives**: Count of active initiatives
- **Dependency Density**: edges / (nodes × (nodes-1))
  - Low density (<0.3): Highly parallelizable
  - High density (>0.7): Highly sequential, limited parallelization
- **Average Path Length**: Mean distance from roots to leaves
- **Longest Chain**: Critical path length
- **Parallelization Factor**: max_parallel_initiatives / total_initiatives

**Storage**: Save graph to `.ibl/dependency_graph.json` for persistence
